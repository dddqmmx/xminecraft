# xminecraft

> **⚠️ PROMINENT WARNING: This is a whimsical project made on a whim, entirely generated by AI.**

`xminecraft` is a Rust TCP tunnel that carries VLESS over TLS inside a Minecraft Server connection. The proxy emulates a Minecraft 1.20.1 server. The outer connection negotiates a standard Minecraft handshake, executes an RSA authentication, and establishes the native Minecraft AES-128-CFB8 encrypted stream. Once the native stream is established, `xminecraft` tunnels TLS and VLESS inside the AES channel.

The project uses `valence_protocol` for Minecraft VarInt and packet field encoding/decoding. It acts as a pair of cooperating endpoints.

## Topology

```text
local SOCKS5/HTTP/TCP client -> xminecraft proxy client 
                             -> Minecraft Login Handshake (RSA + AES-128-CFB8 setup)
                             -> Native Minecraft AES-128-CFB8 Stream
                             -> TLS 1.3
                             -> VLESS Protocol
                             -> xminecraft proxy server 
                             -> target public TCP service
```

## Architecture & Theory of Operation

The proxy stack is built as a series of nested layers:

### 1. Minecraft Layer
The outermost layer of the server implements a Minecraft 1.20.1 protocol stack. When entities connect, they interact with a Minecraft server protocol.
- **Server List Ping:** The server returns MOTD, icon, and latency responses, but hides the player list.
- **Login Handshake & Whitelisting:** The server challenges clients with a Minecraft ServerID and a 1024-bit RSA public key. The server enforces a UUID/Username whitelist. Any unauthorized connection is met with a standard `Login Disconnect` packet and immediately closed.

### 2. AES-128-CFB8 Layer
Once the client successfully passes the whitelist and the RSA shared secret exchange, both ends enable Minecraft's native `AES-128-CFB8` stream cipher.

### 3. TLS 1.3 Layer
`xminecraft` runs a `rustls` TLS 1.3 connection inside the established Minecraft AES stream. This provides Perfect Forward Secrecy (PFS), authenticated encryption, and Server Identity validation.

### 4. VLESS Layer
Inside the TLS tunnel, the client transmits a VLESS request payload containing the user's UUID and the target destination (Domain, IPv4, IPv6).

### 5. Proxy Layer (SOCKS5 / HTTP Proxy)
The local `xminecraft` client acts as an ingress gateway. When an external application connects to the local port, the proxy sniffs the first byte of the TCP stream (`0x05` for SOCKS5, `0x43`/'C' for HTTP CONNECT). It then negotiates the respective protocol, parses the target address, and initiates the tunnel buildup. The local proxy replies with `200 Connection Established` or SOCKS5 `SUCCESS` after the remote tunnel (Minecraft -> TLS -> VLESS) has been successfully established.

## Example

Create or obtain a TLS certificate for the tunnel server name, and choose a
VLESS UUID shared by both endpoints:

```bash
uuidgen
```

Start the server-side tunnel endpoint. It accepts traffic on port `25565`,
terminates TLS, authenticates the VLESS UUID, and then
connects to the target address supplied by the client VLESS request:

```bash
cargo run -- \
  --vless-id 5783a3e7-e373-51cd-8642-c83782b807c5 \
  server \
  --listen 0.0.0.0:25565 \
  --tls-cert fullchain.pem \
  --tls-key privkey.pem \
  --whitelist-file whitelist.json \
  --brand xminecraft \
  --motd "A Minecraft Server"
```

Start the client-side endpoint. It exposes a local TCP port, validates the
server certificate, writes the VLESS TCP request, and then proxies bytes:

```bash
cargo run -- \
  --vless-id 5783a3e7-e373-51cd-8642-c83782b807c5 \
  client \
  --listen 127.0.0.1:1080 \
  --tunnel example.com:25565 \
  --tls-ca-cert ca.pem \
  --tls-server-name example.com
```

By omitting the `--target` parameter, the client automatically acts as a **SOCKS5 and HTTP CONNECT proxy** on port 1080.
It uses protocol sniffing (peeking the first byte) to dynamically route your traffic to any destination:

```bash
# Use as a HTTP Proxy
curl -x http://127.0.0.1:1080 https://stranger-than-heaven.com

# Use as a SOCKS5 Proxy
curl -x socks5h://127.0.0.1:1080 https://stranger-than-heaven.com
```

If you prefer a static port-forwarding mode instead of a dynamic proxy, you can explicitly set `--target`:
```bash
cargo run --client ... --target 127.0.0.1:22
ssh -p 1080 user@127.0.0.1
```

## Security

The data protocol inside the Minecraft AES stream is VLESS over TLS:

- TLS provides confidentiality, integrity, replay protection at the record
  layer, and server identity validation through `--tls-ca-cert` plus
  `--tls-server-name`.
- VLESS provides client authentication with `--vless-id` or the
  `XMINECRAFT_VLESS_ID` environment variable.
- The Minecraft login phase can enforce a whitelist before the TLS/VLESS tunnel
  is opened, using the standard `whitelist.json` shape or explicit CLI entries.
- Do not use `--tls-ca-cert` with an untrusted certificate, and do not set
  `--tls-server-name` to a name that is not covered by the certificate SAN.

## Important Options

```bash
--no-preamble
```

Disables the small Minecraft handshake/login-start preamble between the two
`xminecraft` endpoints. When disabled, the endpoints bypass the Minecraft authentication phase and communicate raw TLS data directly over TCP (not recommended for evasion).

```bash
--vless-id 5783a3e7-e373-51cd-8642-c83782b807c5
```

Shared VLESS UUID. The server rejects requests with a different UUID.

```bash
client --target host:port
```

Target TCP endpoint encoded into the VLESS request. Domain names, IPv4, and
bracketed IPv6 targets are supported.

```bash
client --tls-ca-cert ca.pem --tls-server-name example.com
server --tls-cert fullchain.pem --tls-key privkey.pem
```

TLS settings. The client validates the server certificate chain against the CA
file and verifies the configured server name.

```bash
server --whitelist-file whitelist.json
server --whitelist-user Steve --whitelist-uuid b50ad385-829d-3141-a216-7e7d7539ba7f
```

Enables Minecraft login whitelist checks. The JSON file follows the usual
Minecraft format:

```json
[
  {
    "uuid": "b50ad385-829d-3141-a216-7e7d7539ba7f",
    "name": "Notch"
  }
]
```

When a whitelist is configured, non-matching login attempts receive a
`Login Disconnect` packet with `--whitelist-message` and never reach TLS/VLESS
handling. If no whitelist source is configured, login remains open.

```bash
server --brand xminecraft --entity-id 1 --spawn-y 64
```

Controls the server brand payload, the local player entity id, and the initial
spawn height used in the one-time play-state bootstrap packets.

## Minecraft Protocol Behavior

The server endpoint handles the standard handshake state split:

- `status`: replies with a normal server-list JSON response and echoes ping.
- `login`: checks the optional whitelist, and sends `Login Disconnect` for rejected users. For accepted users, it successfully negotiates the RSA key exchange and transitions into the AES-128-CFB8 encrypted state.
- `play bootstrap`: Immediately after AES encryption begins, the server sends a one-time sequence of play-state packets (feature flags, `Join Game` with a minimal Overworld registry codec, server brand, difficulty, view distances, abilities, spawn/position, health, world time). **There is no ongoing game simulation or game tick loop.** Once this static bootstrap is sent, the AES stream is exclusively handed over to the TLS/VLESS proxy layer.

This keeps the endpoint protocol-correct for the initial connection phase of the Valence-targeted Minecraft version (`1.20.1`, protocol `763`), sufficiently satisfying external active probes while minimizing unnecessary overhead.

## Code Layout

```text
src/cli.rs                command-line parsing and runtime config assembly
src/protocol.rs           Minecraft VarInt packet framing and AES IO wrappers
src/minecraft/            status/login/play bootstrap and active probe defense
src/proxy/                client/server listeners, protocol sniffers (SOCKS5/HTTP), TCP relay
src/tls.rs                rustls client/server configuration
src/vless/                VLESS IDs, target parsing, request/response wire IO
src/whitelist.rs          Minecraft whitelist loading and matching
```

The larger protocol areas are directory modules so data types, wire encoding,
session handling, and tests stay close to their own responsibility instead of
accumulating in a single file.
